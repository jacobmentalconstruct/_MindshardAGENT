0001: # MindshardAGENT
0002: 
0003: MindshardAGENT is a local-first desktop agent workbench for advanced project
0004: building, inspection, and controlled tool use.
0005: 
0006: It combines:
0007: - local Ollama-backed chat and loop execution
0008: - sandbox-aware CLI tool access
0009: - session persistence and project state
0010: - prompt/workbench inspection surfaces
0011: - Prompt Lab foundations for deeper prompt/execution design work
0012: 
0013: This is not a toy chatbot shell anymore. It is closer to a power tool for local
0014: agentic development.
0015: 
0016: ## Who This Is For
0017: 
0018: MindshardAGENT is meant for users who want a serious local agent workbench and
0019: are comfortable operating with project structure, runtime state, and tooling
0020: surfaces that can materially affect files and workflows.
0021: 
0022: If you want a low-risk general chat app, this is probably not the right tool.
0023: 
0024: ## Practical Safety Note
0025: 
0026: MindshardAGENT can be used safely, but it should be used deliberately.
0027: 
0028: Like a table saw or skill saw, the danger is not that the tool is "bad." The
0029: danger is that it is powerful enough to do real work quickly, including work
0030: you may not actually want if it is misconfigured or used carelessly.
0031: 
0032: Examples of what that means in practice:
0033: - agent-driven tool calls can change files inside the active project
0034: - project settings and prompt/runtime configuration can influence behavior in
0035:   non-obvious ways
0036: - experimentation without review can damage a working project state
0037: - draft or testing workflows should not be treated casually on important code
0038: 
0039: Recommended operating habits:
0040: - use version control or backups before major experimentation
0041: - learn on disposable or low-stakes projects first
0042: - review tool-facing behavior before trusting it on important work
0043: - treat configuration changes as real engineering changes, not harmless toggles
0044: - keep prompt/design experiments isolated until they are ready to promote
0045: 
0046: The app is intended to help capable builders move faster, not to remove the
0047: need for judgment.
0048: 
0049: ## Current Capabilities
0050: 
0051: - chat with locally running Ollama models
0052: - execute CLI commands inside a sandbox-aware project context
0053: - stream responses with runtime/status visibility
0054: - save and reload sessions
0055: - inspect runtime activity in a terminal-style log panel
0056: - work with multiple loop modes, including planning-oriented flows
0057: - use a Prompt Workbench inside the app for source/build inspection

0001: # Loop Contract
0002: 
0003: This document defines the execution contract for all loop modules under
0004: `src/core/agent/`.
0005: 
0006: ## Purpose
0007: 
0008: Loops are alternate execution modes selected by the loop manager. They may vary
0009: in internal strategy, but they must present a common external contract so the
0010: app, engine, bridge, benchmarks, and history persistence do not need
0011: loop-specific logic.
0012: 
0013: ## Request Contract
0014: 
0015: Every loop accepts a `LoopRequest` with:
0016: 
0017: - `user_text`: the current user turn text
0018: - `chat_history`: prior persisted chat messages
0019: - `on_token`: optional token streaming callback
0020: - `on_complete`: required completion callback for successful or stopped runs
0021: - `on_error`: callback for unrecoverable loop failure
0022: - `on_tool_start`: optional callback for tool-call display
0023: - `on_tool_result`: optional callback for rendered tool results
0024: - `mode_hint`: optional explicit loop override
0025: 
0026: Loops may read only what they need from the request. They must not mutate the
0027: request in place.
0028: 
0029: ## Completion Contract
0030: 
0031: Successful completion, partial completion, and user-requested stop all report
0032: through `on_complete(result)`. The result payload must contain:
0033: 
0034: - `content`: assistant-visible final text for the turn
0035: - `metadata`: dict containing, at minimum:
0036:   - `loop_mode`
0037:   - `stopped`
0038:   - `rounds`
0039: - `history_addition`: the persisted history rows for this turn
0040: 
0041: Optional fields such as `prompt_build` may be included when the loop owns them.
0042: 
0043: ## History Ownership
0044: 
0045: Loops own their `history_addition` payload. Wrappers that change visible output
0046: must also update `history_addition` to match the final content. Wrappers that do
0047: not change visible output may preserve the wrapped history and only override
0048: metadata such as `loop_mode`.
0049: 
0050: ## Stop Semantics
0051: 
0052: - `request_stop()` must be idempotent.
0053: - A user stop is not an error.
0054: - When a loop stops after beginning work, it should still complete via
0055:   `on_complete(...)` with `metadata.stopped = True`.
0056: - `on_error(...)` is reserved for actual failure, not cancellation.
0057: 
0058: ## Join Semantics
0059: 
0060: Loops that spawn or delegate to background work should expose `join(timeout)`
0061: so shutdown can drain cleanly. Wrapper loops that delegate to another joinable
0062: loop should forward `join(timeout)` to the wrapped loop.
0063: 
0064: ## Allowed Side Effects
0065: 
0066: Loops may:
0067: 
0068: - emit activity-stream messages
0069: - stream tokens via `on_token`
0070: - trigger tool callbacks when they truly own tool execution
0071: - include loop-specific metadata
0072: 
0073: Loops may not:
0074: 
0075: - directly persist session history outside the completion payload
0076: - mutate global selection policy
0077: - silently change another loop's contract

0016:

0018: - `src/app.py`

0036: ## 2. Initial Runtime State
0037:

0039: - a selected model context

0041: - session store and knowledge store bindings

0043: - Prompt Lab summary state

0047: ## 3. Project Attach Flow
0048: 
0049: Project attach is one of the most important runtime transitions.
0050: 
0051: The operator selects a project folder.
0052: Then the system:
0053: - updates `config.sandbox_root`
0054: - rebinds the engine sandbox
0055: - loads or creates project metadata
0056: - updates window/project labels
0057: - reinitializes session and knowledge stores
0058: - creates a new session context
0059: - refreshes prompt inspector state

0063: - `src/core/project/project_command_handler.py`

0066: 
0067: The app can also create and attach a self-edit working copy.
0068: 
0069: That flow:
0070: - copies the source into a working destination
0071: - attaches the resulting sandbox/project
0072: - marks the project with a self-edit profile
0073: - refreshes runtime state similarly to normal project attach
0074: 
0075: This is also coordinated from:
0076: - `src/core/project/project_command_handler.py`
0077: 
0078: ## 5. Prompt Submission Flow
0079: 
0080: When the user submits text in the main app:
0081: - the UI hands it to the app layer
0082: - the app layer routes it into the engine
0083: - the engine builds a loop request
0084: - the loop manager selects or honors the execution path
0085: - the active loop runs
0086: - tool calls and tokens stream back through callbacks
0087: - results are appended to runtime history
0088: - UI updates are posted through the app/UI layers
0089: 
0090: Major files involved:
0091: - `src/app_streaming.py`
0092: - `src/core/engine.py`
0093: - `src/core/agent/loop_manager.py`
0094: - `src/core/agent/loop_selector.py`
0095: - `src/core/agent/response_loop.py`
0096: - specific loop implementations under `src/core/agent/`
0097: 
0098: ## 6. Loop Execution
0099: 
0100: The runtime uses multiple loop types, including:
0101: - direct chat
0102: - planner only
0103: - thought chain
0104: - review judge
0105: - recovery agent
0106: 
0107: The loop family is governed by:
0108: - `src/core/agent/loop_contract.md`
0109: - `src/core/agent/loop_types.py`
0110: 
0111: The important lifecycle idea is that loops are not ad hoc.
0112: They run through a contract for:
0113: - request shape
0114: - result shape
0115: - history ownership
0116: - stop semantics
0117: - metadata
